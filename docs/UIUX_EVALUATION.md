## 2. 良い点（Strengths）

### 2.1 情報設計

- **プログレッシブ・ディスクロージャ**: 設定項目をドロワー内に格納し、さらに詳細設定をアコーディオンで折りたたむことで、メイン画面をシンプルに保っている（`SettingsDrawer.tsx` → `SettingsForm.tsx`）
- **明確な画面構成**: メイン画面は「ヘッダー → 解析制御 → フィードバック表示 → 詳細情報」の一直線レイアウトで、上から下への自然な操作フローを実現（`app/page.tsx`）
- **状態の即座の反映**: Jotaiによるアトミックな状態管理で、設定変更が即時にUI全体へ反映される

### 2.2 ビジュアルデザイン

- **音名カラーマップ**: 12音それぞれに一貫した色を割り当て、全フィードバック表示で統一使用（`lib/constants.ts` の `PITCH_COLOR_MAP`）
- **3段階カラーフィードバック**: 合致（緑 `#22c55e`）、近い（黄 `#eab308`）、外れ（赤 `#ef4444`）の直感的な色分け
- **グラスモーフィズム**: 設定ボタンに半透明+ブラー効果（`bg-background/50 backdrop-blur-sm`）でモダンなUI
- **ダークモード**: OKLCH色空間を使った本格的なダークモード対応（`globals.css`）

### 2.3 インタラクション

- **5種類のフィードバック表示**: メーター・バー・サークル・ストロボ・波形と多様な視覚化モードを選択可能（`UnifiedFeedback.tsx`）
- **ホールド機能**: 検出ロスト時に250ms保持 → 1.5s後フェードアウトで、フリッカーを防止（`METER_NEEDLE_HOLD_MS`, `METER_REMAIN_MS`）
- **滑らかなアニメーション**: メーター針の `transition: transform 0.2s ease-out`、バーの `transition-all duration-150` など自然な動き
- **確認ダイアログ**: プリセット削除・上書きに確認ダイアログを設置し、誤操作を防止（`PresetSaveDialog.tsx`）
- **マイク入力による自動検出**: ピッチ検出ボタンで音名を自動入力でき、手動入力の手間を削減（`MicInputButton.tsx`）

### 2.4 技術的な良い点

- **PWA対応**: Service Worker + マニフェストによるオフライン動作・インストール対応
- **設定の永続化**: localStorageによる全設定の自動保存・復元
- **実験モード**: CSV出力、スペクトルデバッグ表示など研究用途に配慮した機能
- **Zod + React Hook Form**: 型安全なフォームバリデーション
- **全95テスト合格**: コアロジック（純正律計算、根音推定、EMAスムージング）と主要コンポーネントのテストが充実

## 3. 問題点と改善提案

### 3.1 アクセシビリティ（CRITICAL）

#### 3.1.1 ユーザースケーリング無効化

**重要度**: CRITICAL
**場所**: `app/layout.tsx:28-34`

```typescript
export const viewport = {
  width: "device-width",
  initialScale: 1,
  maximumScale: 1,
  userScalable: false,  // ← WCAG 2.1 Level AA 違反
  themeColor: "#3b82f6",
};
```

**問題**: `userScalable: false` はWCAG 2.1 成功基準 1.4.4（テキストのサイズ変更）に違反する。視覚障害のあるユーザーがピンチズームで拡大できない。

**改善案**:
```typescript
export const viewport = {
  width: "device-width",
  initialScale: 1,
  maximumScale: 5,
  userScalable: true,
  themeColor: "#3b82f6",
};
```

#### 3.1.2 タッチターゲットサイズ不足

**重要度**: HIGH
**場所**: `components/feature/PitchList.tsx:59-71`

構成音リストの削除ボタンが `h-5 w-5`（20×20px）で、推奨される最小タッチターゲット44×44pxを大きく下回る。

```tsx
<Button
  variant="ghost"
  size="icon"
  className="h-5 w-5 rounded-full p-0"  // 20×20px
>
  <X className="h-3.5 w-3.5" />
</Button>
```

**改善案**: パディングを追加して実質的なタッチ領域を確保する。

```tsx
<Button
  variant="ghost"
  size="icon"
  className="h-8 w-8 rounded-full p-1"  // 32×32px + hit area
  aria-label={`${data.pitchName}${data.octaveNum} を削除`}
>
  <X className="h-3.5 w-3.5" />
</Button>
```

#### 3.1.3 Canvas/SVG要素のアクセシビリティ欠如

**重要度**: HIGH
**場所**: 各フィードバックコンポーネント

| コンポーネント | 問題 |
|:---|:---|
| `WaveformFeedback.tsx` | `<canvas>` に `role="img"` と `aria-label` なし |
| `CircleFeedback.tsx` | `<svg>` にアクセシブルな説明なし |
| `MeterFeedback.tsx` | `<svg>` にアクセシブルな説明なし |
| `StroboFeedback.tsx` | ストロボ表示にテキスト代替なし |

**改善案**: 各ビジュアル要素に `role="img"` と動的な `aria-label` を付与する。

```tsx
<canvas
  ref={canvasRef}
  role="img"
  aria-label={`${pitchName}のセント偏差: ${deviation.toFixed(1)}セント`}
/>
```

#### 3.1.4 色のみに依存した情報伝達

**重要度**: MEDIUM
**場所**: `CircleFeedback.tsx:132`, `BarFeedback.tsx:74-78`

合致/不合致の判定が緑/赤の色のみで表現されており、色覚異常のユーザーには区別困難。

**改善案**: 色に加えてテクスチャ（パターン）やアイコン（✓/×）を併用する。BarFeedbackでは既に一部テキストラベル（"低い"/"高い"）があるが、全フィードバックで統一すべき。

### 3.2 レスポンシブ対応

#### 3.2.1 メーター表示の固定幅

**重要度**: HIGH
**場所**: `components/feedback/MeterFeedback.tsx:119`

```tsx
<div className="flex w-64 h-32 overflow-hidden mb-4">
```

`w-64`（256px）は固定値であり、320px幅の小型スマートフォンで余白がほぼなくなる。

**改善案**: `w-full max-w-64` に変更してレスポンシブ対応する。

#### 3.2.2 CentDisplay テーブルのモバイル対応

**重要度**: MEDIUM
**場所**: `components/feature/CentDisplay.tsx:156-217`

テーブルに「音名」「音程」「セント差」「再生（2ボタン）」の4列があり、小画面で横方向にはみ出す可能性がある。

**改善案**: モバイルではカード型レイアウトに切り替えるか、テーブルに `overflow-x-auto` を適用する。

#### 3.2.3 設定ドロワーの小画面対応

**重要度**: LOW
**場所**: `components/layout/SettingsDrawer.tsx:49`

ドロワー幅 `max-w-sm`（384px）は適切だが、内部のフォーム要素が狭い画面で窮屈になる場合がある。特に `PitchSettingForm` のセレクトボックス群が横並びの場合。

---

### 3.3 エラーハンドリング

#### 3.3.1 ブラウザ alert() の使用

**重要度**: MEDIUM
**場所**: `lib/hooks/useAudioAnalysis.ts:64`

```typescript
alert("マイクへのアクセスを許可してください。");
```

`alert()` はモーダルダイアログを強制し、UXが低下する。また、PWAやフルスクリーン表示時に不適切。

**改善案**: Toast通知またはインラインのエラーメッセージに置き換える。shadcn/uiの `Sonner` や `Toast` コンポーネントの導入を推奨。

#### 3.3.2 空状態のガイダンス不足

**重要度**: MEDIUM
**場所**: `components/feedback/UnifiedFeedback.tsx:53-59`

```tsx
<div className="text-center py-8 text-muted-foreground">
  解析データがありません
</div>
```

単にデータがないことを表示するだけで、次のアクションへの導線がない。

**改善案**: 「設定から構成音を追加して、解析を開始してください」のようなアクション指示 + 設定ドロワーを開くボタンを追加。

#### 3.3.3 解析中のローディング表示

**重要度**: LOW
**場所**: `app/page.tsx:92-96`

解析中のステータス表示がテキストのみ（`マイク入力からの解析中...`）で、視覚的インジケーターがない。

**改善案**: スピナーアイコンやプログレスバーを追加して、処理中であることを明確にする。

---

### 3.4 ダークモードの不完全性

#### 3.4.1 SVG内のハードコードカラー

**重要度**: MEDIUM
**場所**: `components/feedback/CircleFeedback.tsx:109-115, 149, 154, 179, 189, 207`

```tsx
<circle cx="300" cy="300" r="150" fill="#f0f0f0" stroke="#333" strokeWidth="2" />
// テキスト
<text fill="#000000" ...>
```

ダークモードでは `#f0f0f0`（薄灰色）が暗い背景に溶け込み、`#000000`（黒）のテキストも見えにくくなる。

**改善案**: CSS変数またはTailwindのダークモードクラスを使用して動的に色を切り替える。

---

### 3.5 パフォーマンス

#### 3.5.1 Canvas再描画の最適化

**重要度**: LOW
**場所**: `components/feedback/WaveformFeedback.tsx:18-78`

```tsx
useEffect(() => {
  // canvas描画処理
}, [deviation, color]);
```

`deviation` が高頻度で更新されるため、毎回Canvasを全面再描画する。低スペックデバイスでは負荷が高い可能性がある。

**改善案**: `requestAnimationFrame` でフレームレートを制御するか、描画をスロットリングする。

## 4. ユーザーフロー評価

### 4.1 メインフロー: 和音チューニング

```
[設定を開く] → [音名選択] → [オクターブ選択] → [根音チェック] → [追加]
    → [解析開始] → [フィードバック確認] → [楽器調整] → [解析停止]
```

**評価**: フローは論理的で迷いにくい。ただし、初回利用時に「何をすればよいか」のオンボーディングがなく、設定ドロワーを開く必要があることが分かりにくい。

**改善提案**:
- 初回アクセス時のチュートリアルまたはウォークスルー
- 構成音未追加時にメイン画面にCTA（Call to Action）を表示

### 4.2 プリセットフロー

```
[構成音を設定] → [プリセット保存] → [名前入力] → [保存確認]
    → [次回: プリセット読み込み] → [確認] → [構成音復元]
```

**評価**: 保存・読み込み・削除の各操作に確認ダイアログがあり、安全性が高い。プリセットカードに保存日時（相対時間）が表示されるのも良い。文字数制限（30文字）のリアルタイム表示も適切。

### 4.3 実験モードフロー

```
[設定 → 実験モード ON] → [黄色バナー表示] → [ログ記録開始]
    → [演奏・解析] → [ログ記録停止] → [CSVエクスポート]
```

**評価**: 通常モードと実験モードの切り替えが明確。バナーとステータスインジケーター（赤い点滅）で現在の状態が把握しやすい。

## 7. 総括

ChordLensは、純正律に基づいた和音チューニングという専門性の高い領域において、直感的で多機能なWebアプリケーションを実現している。5種類のフィードバック表示、プリセット管理、実験モードなど、研究ツールとしての完成度は高い。

**最優先で改善すべき項目**:
1. **`userScalable: false` の撤回** — WCAG違反であり、1行の変更で修正可能
2. **タッチターゲットサイズの拡大** — モバイルユーザーの操作性に直結
3. **ビジュアル要素のアクセシビリティ** — aria-label追加で支援技術への対応

これらはいずれも小〜中程度の工数で対応でき、ユーザー体験を大きく改善できる。
